Teledot

Role: Solo developer (design, implementation, UI/UX, release)
Tools: Construct 3
Status: Shipped playable web build on itch.io; app store launch planned but paused
Timeline: Weekend-scale build (shipped)

Context

Teledot is a minimalist reflex survival game built to prove end-to-end execution: rapid engine onboarding, clean mechanics, strong game-feel, and a shipped public build. The design deliberately emphasizes clarity and polish over feature count.

Problem

As a first shipped game, the goal was to avoid common beginner failure modes: over-scoping, unstable logic, and incomplete releases. The project needed to be small enough to finish, legible within seconds, and polished enough to demonstrate UX/UI taste and gameplay clarity.

Strategy

  • Constrain design to a single high-signal core mechanic (teleport movement)
  • Build a state-driven flow before adding content
  • Prioritize input determinism across desktop and mobile
  • Invest in feedback, pacing, and restart loop clarity

Execution

  • State machine: “TITLE” → “PLAY” → “DEAD” governs gameplay and UI
  • Cross-platform input: mouse + touch with explicit input mode flag for deterministic behavior
  • Teleport movement clarity: clamped positioning and afterimage trail for readable motion
  • Spawner architecture: accumulator-based hazard spawning tied to dt for stable pacing
  • Difficulty scaling: survival time drives spawn rate growth
  • Death sequence polish: slow-motion time scale, timed UI reveals, restart lockout
  • Audio UX: mute toggle, track skip, auto-advance on track end
  • Visual modulation: periodic warp ripple pulses controlled via layer effect parameters

Outcome

  • Delivered a complete, playable build publicly on itch.io
  • Established a repeatable pattern for future projects: scope hardening → state machine first → polish pass → ship
  • Used Teledot as a baseline to move into more complex games while maintaining execution standards

Next Steps

  • Package for mobile app stores once release pipeline becomes the priority
  • Add optional monetization and content expansion (skins/variants/modifiers) without changing core loop
  • Add lightweight mobile settings/accessibility pass (touch sensitivity, haptics, safe area)
